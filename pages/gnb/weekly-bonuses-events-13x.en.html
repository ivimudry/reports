Weekly Bonuses Event 13.x (EN)

Chill4Reel — Weekly Bonuses Event 13.x (EN)

Purpose: define a single recurring event model for Weekly Bonuses campaigns 13.1–13.7. All weekly bonus sends use one event, weekly_bonuses. The specific weekly campaign is selected by the event attribute bonus_type.

0. Executive summary

The Weekly Bonuses block should use one shared event trigger for all weekday variants. A scheduler sends weekly_bonuses to all existing users on the configured weekly schedule. The event carries one required attribute: bonus_type.

Core rule: the scheduler does not decide eligibility. It sends the event to all existing users. The chain then filters users by bonus_type, global suppressions, active bonus state, consent, channel availability, and campaign-specific rules.

1. Event logic

All 13.x weekly campaigns use the same operating model:

  1. The scheduler emits weekly_bonuses on the configured weekday and time.
  2. The event includes bonus_type to identify the weekly campaign variant.
  3. The event is sent to all existing users in the workspace.
  4. The campaign chain listens to that event as its trigger.
  5. The first branch inside the chain checks bonus_type and routes users to the matching 13.x campaign path.
  6. The chain then applies all eligibility and suppression checks.
  7. Only users who pass the chain filters continue to receive campaign messages.

Important: monday_weekly, tuesday_weekly, and the other weekday values are not separate event names. They are values of bonus_type inside the shared weekly_bonuses event.

4. Minimal event payload

The only required event attribute is bonus_type. Additional metadata is optional and should be added only if it is needed for reporting or debugging.

Field Type Required Description

scheduled_for timestamp No Optional scheduler timestamp if needed for reporting
source string No Scheduler name or system that emitted the event

5. Chain filtering model

The shared weekly_bonuses event should be treated as a broad trigger. The chain must immediately route and filter users after the event is received.

Filter layer Where it happens Purpose
Weekly route Inside campaign chain Continue only through the path matching event.bonus_type
Global suppression Inside campaign chain Exclude self-excluded, cool-off, invalid email, non-consented, blocked, duplicated, internal/non-production, dormant archive, or bonus-abuser users as required
Bonus conflict check Inside campaign chain Prevent users with an active incompatible bonus from receiving a new promo
Campaign-specific eligibility Inside campaign chain Apply rules specific to the weekly offer, such as deposit state, loyalty state, or previous interaction state
Channel eligibility Inside campaign chain Apply Email, Push, SMS, or In-App consent and availability checks before each message
Frequency control Inside campaign chain Prevent duplicate sends from the same weekly event batch where needed

Implementation note: the scheduler should not pre-filter the audience. It should only send weekly_bonuses with the correct bonus_type. All business logic stays inside CIO.
