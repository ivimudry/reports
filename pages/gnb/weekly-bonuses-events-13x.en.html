Weekly Bonuses Events 13.x (EN)

Chill4Reel — Weekly Bonuses Events 13.x (EN)

Purpose: define the event-based trigger model for Weekly Bonuses campaigns 13.1–13.7. Each campaign receives its own scheduled weekly event. The event is sent to all existing users, and all eligibility, suppression, bonus-state, consent, and campaign-specific checks are handled inside the chain.

0. Executive summary

The current Weekly Bonuses block can be moved to a simple event-triggered model. Instead of relying on manually targeted sends or separate audience pushes per weekly promo, a scheduler emits one dedicated event for each campaign on its fixed weekly cadence.

Core rule: the scheduler does not decide who should receive the message. It sends the weekly event to all existing users. The campaign chain then filters users through its own conditions before sending Email, Push, SMS, In-App, or any follow-up step.

1. Event logic

All 7 weekly campaigns use the same operating model:

  1. The scheduler emits one dedicated event for the specific weekly campaign.
  2. The event is sent to all existing users in the workspace.
  3. The campaign chain listens to that event as its trigger.
  4. The first steps inside the chain apply all eligibility and suppression checks.
  5. Only users who pass the chain filters continue to receive campaign messages.
  6. The event repeats every week on the same weekday and time.

Important: event delivery and audience qualification are separate. The scheduler sends the event broadly; the chain decides who is allowed to continue.

3. Standard weekly payload

Each event should use the same payload shape. The event name identifies the campaign, while the payload gives enough context for reporting, debugging, and downstream Liquid logic if needed.

Field Type Required Description
campaign_id string Yes Campaign number, for example 13.1, 13.2, 13.7
campaign_name string Yes Human-readable campaign name, for example Monday Happy Spins
weekly_event string Yes Same value as the event name, for example monday_weekly
scheduled_for timestamp Yes Scheduled send timestamp in the agreed campaign timezone
weekday string Yes Calendar day used by the scheduler, for example sunday, tuesday, wednesday
weekly_batch_id string Yes Unique ID for one weekly send batch; used for dedupe and reporting
source string No Scheduler name or system that emitted the event

4. Chain filtering model

Every 13.x campaign should treat the weekly event as a broad trigger, then immediately apply its own filters inside the chain.

Filter layer Where it happens Purpose
Global suppression Inside campaign chain Exclude self-excluded, cool-off, invalid email, non-consented, blocked, duplicated, internal/non-production, dormant archive, or bonus-abuser users as required
Bonus conflict check Inside campaign chain Prevent users with an active incompatible bonus from receiving a new promo
Campaign-specific eligibility Inside campaign chain Apply rules specific to the weekly offer, such as deposit state, loyalty state, or previous interaction state
Channel eligibility Inside campaign chain Apply Email, Push, SMS, or In-App consent and availability checks before each message
Frequency control Inside campaign chain Prevent duplicate sends from the same weekly event batch where needed

Implementation note: the scheduler should not pre-filter the audience. Keeping the scheduler broad makes the weekly event contract simple and moves all business logic into CIO, where campaign behavior is visible and editable.

5. Scheduling rules

Rule Definition
Audience All existing users; campaign-level filtering happens inside the chain after the event is received
Delivery model One event push per campaign, per week, sent to every existing user profile
Recurrence Weekly, same weekday and time
Event ownership Backend scheduler or CIO-compatible orchestration layer
Filtering ownership CIO campaign chain
