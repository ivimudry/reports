active_player_period — Customer.io attribute

What it is: the time-of-day period when a player most often logs in to the casino. Used by marketing to send messages at the player's active hours.

Data source: casino login events (session.start_time or equivalent — the successful login event). Session duration, bets, deposits — not used. Only the login event itself.

Algorithm

  1. Take the player's last 30 casino logins (field session.start_time, UTC).
  2. If logins < 5 or the player has no timezone → unknown, exit.
  3. Convert each start_time to the player's local time, keep only the hour (0–23).
  4. Each login awards points to buckets:
    • +1.0 to its own bucket;
    • +0.5 to the previous bucket;
    • +0.5 to the next bucket.
    Buckets wrap around: after night comes late_night again.
  5. Winner — the bucket with the highest total score. That is active_player_period.
  6. If several buckets are tied — see the "Tie-break" section below.
  7. Push the result to Customer.io via API.

Tie-break (when scores are equal)

  1. Compare "raw" logins. The bucket with more actual logins (without neighbour bonuses) wins.
  2. If still tied — fixed period priority:
    1. evening
    2. night
    3. afternoon
    4. morning
    5. early_morning
    6. late_night
    Rationale: evening/night is the peak casino traffic time — a safe default for messaging.
  3. If still tied (very unlikely) → unknown.
Why "+0.5 to neighbours": a login at 15:55 formally falls into afternoon, but it's almost evening. Neighbours get half a point so bucket boundaries don't distort the result. If the player has one clear period — it still wins. If activity is spread between two adjacent periods — the closer one wins.
Important: buckets are circular. The neighbour to the left of late_night is night; the neighbour to the right of night is late_night. Otherwise a player logging in at 23:00 and 01:00 will get garbage.

Hour → bucket mapping

bucket_index = floor(hour / 4)
0 → late_night
1 → early_morning
2 → morning
3 → afternoon
4 → evening
5 → night

Python

BUCKETS = ['late_night', 'early_morning', 'morning',
           'afternoon', 'evening', 'night']
PRIORITY = ['evening', 'night', 'afternoon',
            'morning', 'early_morning', 'late_night']

def active_period(hours):
    """hours — list of local hours of casino logins (0..23)."""
    if len(hours) < 5:
        return 'unknown'

    score = [0.0] * 6
    raw = [0] * 6
    for h in hours:
        i = h // 4
        raw[i] += 1
        score[i] += 1.0
        score[(i - 1) % 6] += 0.5
        score[(i + 1) % 6] += 0.5

    max_score = max(score)
    candidates = [i for i, s in enumerate(score) if s == max_score]
    if len(candidates) == 1:
        return BUCKETS[candidates[0]]

    # Tie-break 1: more raw logins
    max_raw = max(raw[i] for i in candidates)
    candidates = [i for i in candidates if raw[i] == max_raw]
    if len(candidates) == 1:
        return BUCKETS[candidates[0]]

    # Tie-break 2: fixed priority
    for name in PRIORITY:
        if BUCKETS.index(name) in candidates:
            return name

    # Tie-break 3: very rare case
    return 'unknown'

# example: logins mostly in the evening
print(active_period([16, 17, 18, 19, 20, 15, 19, 21]))  # evening

SQL (PostgreSQL)

WITH buckets AS (
  SELECT
    user_id,
    FLOOR(EXTRACT(HOUR FROM start_time AT TIME ZONE user_tz) / 4)::INT AS b
  FROM sessions
  WHERE start_time > NOW() - INTERVAL '30 days'
),
counts AS (
  SELECT user_id, b, COUNT(*) AS c
  FROM buckets
  GROUP BY user_id, b
),
-- distribute points: +1 to own bucket, +0.5 to neighbours (circular)
scored AS (
  SELECT user_id, b AS bucket, c::FLOAT AS score FROM counts
  UNION ALL
  SELECT user_id, (b + 1) % 6, c * 0.5 FROM counts
  UNION ALL
  SELECT user_id, (b + 5) % 6, c * 0.5 FROM counts  -- (b - 1 + 6) mod 6
),
raw_counts AS (
  SELECT user_id, b AS bucket, c AS raw FROM counts
),
totals AS (
  SELECT
    s.user_id,
    s.bucket,
    SUM(s.score) AS total,
    COALESCE(MAX(r.raw), 0) AS raw,
    ROW_NUMBER() OVER (
      PARTITION BY s.user_id
      ORDER BY
        SUM(s.score) DESC,                    -- main score
        COALESCE(MAX(r.raw), 0) DESC,         -- tie-break 1: raw logins
        CASE s.bucket                         -- tie-break 2: priority
          WHEN 4 THEN 1  -- evening
          WHEN 5 THEN 2  -- night
          WHEN 3 THEN 3  -- afternoon
          WHEN 2 THEN 4  -- morning
          WHEN 1 THEN 5  -- early_morning
          WHEN 0 THEN 6  -- late_night
        END
    ) AS rn
  FROM scored s
  LEFT JOIN raw_counts r ON r.user_id = s.user_id AND r.bucket = s.bucket
  GROUP BY s.user_id, s.bucket
),
qualified AS (
  SELECT user_id FROM buckets GROUP BY user_id HAVING COUNT(*) >= 5
)
SELECT
  q.user_id,
  CASE t.bucket
    WHEN 0 THEN 'late_night'
    WHEN 1 THEN 'early_morning'
    WHEN 2 THEN 'morning'
    WHEN 3 THEN 'afternoon'
    WHEN 4 THEN 'evening'
    WHEN 5 THEN 'night'
  END AS active_player_period
FROM qualified q
JOIN totals t ON t.user_id = q.user_id AND t.rn = 1;

Schedule

Batch job once a day at 03:00 UTC. Processes all players who had casino logins in the last 30 days, pushes the result to Customer.io via API.
