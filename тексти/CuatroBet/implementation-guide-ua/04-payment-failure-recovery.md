# 04 — Payment Failure Recovery — Гайд з імплементації

**Ланцюг:** Payment Failure Recovery  
**Пріоритет:** Високий (найдешевша ініціатива, найвищий ROI)  
**Очікувані комунікації:** 4 маршрути відмов + 1 святкування = 5 точок контакту  
**Вимога до швидкості:** Усі recovery-попапи мають спрацьовувати протягом 5 СЕКУНД після події відмови  
**Передумови з Master Doc:** Подія `deposit_failed` з полем `failure_reason` (`INSUFF`/`METHOD`/`DECLINE`/`TIMEOUT`); API платіжних методів; Діплінки `{base_url}/cashier?method={method}`, `{base_url}/cashier?amount={amount}`

---

## КРОК 0: ПЕРЕДСТАРТОВИЙ ЧЕК-ЛИСТ

- [ ] Подія `deposit_failed` спрацьовує протягом 5 секунд з точними кодами `failure_reason`
- [ ] Коди причин відмови розрізняються: `INSUFF`, `METHOD`, `DECLINE`, `TIMEOUT`
- [ ] Система in-app попапів може спрацьовувати протягом 5 секунд після події (потрібна субсекундна затримка)
- [ ] API платіжних методів доступне (для показу альтернативних методів)
- [ ] Діплінки підтримують попередній вибір способу оплати: `{base_url}/cashier?method={method_id}`
- [ ] Діплінки підтримують попереднє заповнення суми: `{base_url}/cashier?amount={amount}`
- [ ] Механізм автоматичного повтору доступний для відмов через таймаут
- [ ] Шаблон бонусу 100 CC (Cuatro Coins) готовий для святкування відновлення
- [ ] Передача до ланцюга Failed Deposits (06) налаштована на T+30хв

**⚠️ КРИТИЧНО:** Якщо GR8 Tech не може розрізнити причини відмов (наприклад, усі відмови повертають загальну помилку), логіка маршрутизації руйнується. Узгодьте з командою платформи забезпечення точних кодів причин ДО побудови цього ланцюга.

---

## КРОК 1: СТВОРЕННЯ JOURNEY

**Campaign Name:** `04-PFR-JOURNEY`  
**Тип:** Triggered journey (на основі подій)  
**Тригер входу:** Event = `deposit_failed`  
**Повторний вхід:** Так (гравець може мати кілька відмов)  
**Кулдаун повторного входу:** 5 хвилин (уникнення спаму при швидких повторах)  
**Вікно Journey:** 30 хвилин (після чого → передача до ланцюга 06)

## КРОК 3: ПОБУДОВА КОЖНОГО МАРШРУТУ ВІДМОВИ

### 3.1 Touch 1A — Відновлення при недостатніх коштах

| Налаштування | Значення |
|--------------|----------|
| **Campaign ID** | `04-PFR-01A-POPUP` |
| **Канал** | In-session popup |
| **Час** | Протягом 5 секунд після відмови |

**Контент попапу:**
- **Banner Text (ES):** `Probá con un Monto Menor`
- **Опис банера:** Компактний recovery-попап — іконка гаманця зі стрілкою вниз, дружній візуал «спробуй менше», три попередньо встановлені варіанти сум підсвічені, спокійні сині/зелені тони
- **Body:** Три кнопки з попередньо встановленими сумами відображаються на першому плані
- **Кнопки:**
  | Напис на кнопці | Дія |
  |-----------------|-----|
  | `500 ARS` | Депозит в один клік → `{base_url}/cashier?amount=500` |
  | `1,000 ARS` | Депозит в один клік → `{base_url}/cashier?amount=1000` |
  | `1,500 ARS` | Депозит в один клік → `{base_url}/cashier?amount=1500` |
- **Закриття:** Текстове посилання «Cancelar» під кнопками

**Логіка:**
- Сума, яку намагався внести гравець, ймовірно перевищувала його баланс
- Попередньо встановлені суми — це типові невеликі вхідні суми для аргентинського ринку
- Дія в один клік зменшує тертя — гравцю не потрібно повторно вводити платіжні дані

**Кроки конфігурації:**
1. На гілці A додати вузол попапу
2. Налаштувати попап з 3 CTA-кнопками (кожна — окремий діплінк на депозит з попередньо заповненою сумою)
3. Встановити затримку спрацювання попапу ≤5 секунд
4. Слухати подію `deposit_success` після показу попапу
5. Якщо успіх → Touch 2 (святкування)
6. Якщо повторна відмова або бездіяльність протягом 30 хв → передача до ланцюга 06

### 3.2 Touch 1B — Відновлення при недоступності методу

| Налаштування | Значення |
|--------------|----------|
| **Campaign ID** | `04-PFR-01B-POPUP` |
| **Канал** | In-session popup |
| **Час** | Протягом 5 секунд після відмови |

**Контент попапу:**
- **Banner Text (ES):** `Probá con Otro Método`
- **Опис банера:** Попап зміни методу — іконка платіжної картки з іконками оновлення/обміну, логотипи альтернативних платежів (Mercado Pago, банківський переказ), зелені підтримуючі акценти
- **Body (ES):** `Tu método de pago no está disponible en este momento. Elegí otro:`
- **Кнопки:** Динамічні — показуються 2–3 інші активні платіжні методи, доступні гравцю
  | Кнопка | Дія |
  |--------|-----|
  | `{alt_method_1_name}` | `{base_url}/cashier?method={alt_method_1_id}` |
  | `{alt_method_2_name}` | `{base_url}/cashier?method={alt_method_2_id}` |
  | `{alt_method_3_name}` (якщо є) | `{base_url}/cashier?method={alt_method_3_id}` |
- **Закриття:** Текстове посилання «Cancelar»

**Необхідні дані:**
- Виклик API платіжних методів для отримання альтернатив (виключити метод, що відмовив)
- Зіставлення ID методів з назвами для відображення (напр., «Mercado Pago», «Tarjeta de crédito», «Transferencia»)

**Кроки конфігурації:**
1. На гілці B викликати API платіжних методів для отримання альтернатив
2. Показати попап з динамічними кнопками для кожного доступного методу
3. Кожна кнопка відкриває касу з попередньо обраним методом
4. Сума оригінальної спроби зберігається
5. Слухати `deposit_success` → Touch 2
6. Відмова або таймаут → ланцюг 06

### 3.3 Touch 1C — Відновлення при відхиленні еквайєром

| Налаштування | Значення |
|--------------|----------|
| **Campaign ID** | `04-PFR-01C-POPUP` |
| **Канал** | In-session popup |
| **Час** | Протягом 5 секунд після відмови |

**Контент попапу:**
- **Banner Text (ES):** `Tu Banco Rechazó el Pago`
- **Опис банера:** Попап відхилення банком — іконка банку з позначкою X, символ щита, альтернативні методи показані нижче, спокійні заспокійливі бурштинові тони
- **Body (ES):** `No te preocupes, probá con otro método de pago:`
- **Кнопки:** Такі самі як Touch 1B — динамічні альтернативні методи

**Примітки:**
- Проблема в банку, не в платформі

- Така сама імплементація як 1B, інший текст

**Кроки конфігурації:**
1. На гілці C — те саме що 1B, але з іншим заголовком/текстом body
2. Показати альтернативні платіжні методи
3. Слухати `deposit_success` → Touch 2
4. Відмова або таймаут → ланцюг 06

### 3.4 Touch 1D — Відновлення при таймауті / технічній проблемі

| Налаштування | Значення |
|--------------|----------|
| **Campaign ID** | `04-PFR-01D-POPUP` |
| **Канал** | In-session popup (автоповтор) |
| **Час** | Протягом 5 секунд після відмови |

**Контент попапу — Фаза 1 (Автоповтор):**
- **Banner Text (ES):** `Reintentando...`
- **Опис банера:** Візуал завантаження/повтору — кругова анімація спінера, іконка шестерні, спокійна синя тема обробки, індикатор прогресу
- **Text (ES):** `Reintentando…`
- **Поведінка:** Автоматичний повтор того самого депозиту з тією ж сумою та методом через 10 секунд
- **Дія гравця не потрібна**

**Контент попапу — Фаза 2 (Якщо автоповтор не вдався):**
- **Banner Text (ES):** `Problema Técnico`
- **Опис банера:** Попап технічної проблеми — іконка гайкового ключа/шестерні з попереджувальним трикутником, дві чіткі кнопки дій нижче, підтримуючі бурштиново-сині тони
- **Body (ES):** `Podés reintentar o probar con otro método:`
- **Button 1 (ES):** `Reintentar` → повтор того самого депозиту
- **Button 2 (ES):** `Otro método` → показати альтернативи (як у 1B)
- **Закриття:** Текстове посилання «Cancelar»

**Кроки конфігурації:**
1. На гілці D запустити автоповтор через 10 секунд
2. Показати попап зі спінером та текстом «Reintentando…»
3. Якщо автоповтор вдався → Touch 2 (святкування)
4. Якщо автоповтор не вдався → замінити спінер на попап з ручним повтором + альтернативами
5. Слухати `deposit_success` → Touch 2
6. Без успіху протягом 30 хв → ланцюг 06

## КРОК 5: ПЕРЕДАЧА ДО ЛАНЦЮГА 06 (Failed Deposits)

| Налаштування | Значення |
|--------------|----------|
| **Час** | T+30 хвилин після початкової відмови |
| **Умова** | Немає успішного депозиту у вікні відновлення |

**Кроки конфігурації:**
1. Після 30 хвилин без події `deposit_success` → завершити цей journey
2. Позначити гравця тегами `payment_failed_unrecovered = true` та `last_failure_reason = {reason}`
3. Ланцюг 06 (Failed Deposits) підхоплює з T+5хв зі своєю комунікаційною драбиною

## КРОК 6: ЧЕК-ЛИСТ ТЕСТУВАННЯ

- [ ] Попап спрацьовує протягом 5 секунд після події `deposit_failed` (виміряти затримку)
- [ ] Гілка правильно маршрутизує за кодами причин відмови
- [ ] Touch 1A: Кнопки попередньо встановлених сум (500, 1000, 1500) працюють як депозити в один клік
- [ ] Touch 1B: Альтернативні платіжні методи завантажуються динамічно та коректно
- [ ] Touch 1C: Ті самі альтернативні методи що й 1B, але з іншим текстом
- [ ] Touch 1D: Автоповтор спрацьовує через 10 секунд
- [ ] Touch 1D: Якщо автоповтор не вдався — показується ручний повтор + альтернативи
- [ ] Touch 2: Святкувальний попап з'являється при успішному відновленні
- [ ] Touch 2: 100 CC нараховано на баланс гравця
- [ ] Кулдаун повторного входу (5 хв) запобігає спаму попапів при швидких відмовах
- [ ] Передача до ланцюга 06 на T+30хв для невідновлених відмов
- [ ] Попапи не з'являються, якщо гравець НЕ в сесії (граничний випадок: гравець закриває додаток під час повтору)
- [ ] Обмеження частоти: recovery-попапи ВИКЛЮЧЕНІ з денних лімітів повідомлень (транзакційні)

**Навантажувальний тест:**
- [ ] Симулювати 100 одночасних відмов — усі попапи спрацьовують протягом 5 секунд
- [ ] Немає дублікатів попапів на одну подію відмови
